Perform a systematic crash audit across the following sources and produce `54-TRIAGE.md`:

**Sources to scan:**

1. **TypeScript compile check** — run `npx tsc --noEmit` and log any errors (expected: 0).

2. **Test suite failures** — run `npx vitest run 2>&1` and classify each failure:
   - `tests/pii/ner-filter.acceptance.test.ts` (10 tests): ECONNREFUSED to Ollama port 11434 — document as EXPECTED/by-design (test tagged `requires-ollama`, needs live Ollama instance). Severity: INFO, not a crash.
   - Any other failures: classify as P0/P1/P2.

3. **Runtime crash vectors** — inspect the following known fragile areas from CONCERNS.md:
   - `src/worker.ts`: unhandledRejection handler (lines ~887-891) — verify it logs + exits cleanly; check DDL race at lines 960-980.
   - `src/lib/health/alerts.ts`: in-memory cooldown Map resets on restart → P2 tech debt (not crash).
   - `src/app/api/onlyoffice/callback/route.ts:49-63`: unauthenticated path when no JWT → P1 security defect, classify.
   - `src/lib/pii/ner-filter.ts:20`: hardcoded NER_MODEL "qwen3.5:35b" ignores provider settings → if Ollama offline, embeddings blocked → classify P1 availability risk.
   - `src/worker.ts:960-980`: DDL run at startup (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`) — race condition if two workers start simultaneously → P2.

4. **Silent catch survey** — run: `grep -rn "\.catch\s*(" src --include="*.ts" --include="*.tsx" | grep -v "__tests__\|\.test\.\|\.spec\." | wc -l` — log count; sample the first 20 hits; identify any that swallow errors in critical paths (DB writes, file uploads, email send) vs. fire-and-forget notification paths (acceptable). Classify critical-path silent catches as P1.

5. **Docker healthcheck gaps** — verify docker-compose.yml:
   - `app` service has healthcheck (GET /api/health) — note `start_period: 60s`.
   - `worker` service has NO healthcheck — classify as P1 (worker can silently die, compose reports healthy).
   - All 8 core services (db, redis, minio, meilisearch, onlyoffice, stirling-pdf, ollama, app) have healthchecks — verify each is correct.
   - `stirling-pdf` internal healthcheck: `curl -f http://localhost:8080/api/v1/info/status` (inside container = correct port). App env: `STIRLING_PDF_URL=http://stirling-pdf:8080` (Docker network = correct). Mark as verified.

| Ollama | ailawyer-ollama | `docker inspect ailawyer-ollama --format '{{.State.Health.Status}}'` | healthy |

| Worker | ailawyer-worker | `docker inspect ailawyer-worker --format '{{.State.Running}}'` | true (no healthcheck — see TRIAGE C-01) |

**Aggregate check script** — include a bash one-liner that checks all services at once:
```bash
for c in ailawyer-db ailawyer-redis ailawyer-minio ailawyer-meilisearch ailawyer-onlyoffice ailawyer-stirling-pdf ailawyer-ollama ailawyer-app; do
  status=$(docker inspect $c --format '{{.State.Health.Status}}' 2>/dev/null || echo "not_found")
  echo "$c: $status"
done
# Worker has no healthcheck — check running state separately:
docker inspect ailawyer-worker --format 'worker: {{.State.Running}}' 2>/dev/null
```

4. Worker alive: `curl -s http://localhost:3000/api/health | jq '.status'` → if worker shows unhealthy, check `docker logs ailawyer-worker --tail 50`

Document each test with: Test ID, Description, Command, Expected Result, Failure diagnosis hint.
