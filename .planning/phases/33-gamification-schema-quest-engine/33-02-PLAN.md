**File 1: `src/lib/gamification/quest-evaluator.ts`**

Create the DSL evaluator that translates QuestCondition JSON into Prisma COUNT queries:

1. `evaluateQuestCondition(condition: QuestCondition, userId: string, date?: Date): Promise<{ current: number; target: number; completed: boolean }>`

Implementation:
- Calculate date range from `condition.period` using date-fns:
  - `"today"`: startOfDay(date) to endOfDay(date)
  - `"thisWeek"`: startOfWeek(date, { weekStartsOn: 1 }) to endOfWeek(date, { weekStartsOn: 1 })
  - `"thisMonth"`: startOfMonth(date) to endOfMonth(date)

- Build where clause:
  ```typescript
  const where: Record<string, any> = {
    ...condition.where,
    [condition.dateField]: { gte: rangeStart, lte: rangeEnd },
  };
  if (condition.userField) {
    where[condition.userField] = userId;
  }
  ```

- For Rechnung (userField is null), add a special Akte-relation scope:
  ```typescript
  if (condition.model === "Rechnung" && !condition.userField) {
    where.akte = {
      OR: [
        { anwaltId: userId },
        { sachbearbeiterId: userId },
      ],
    };
  }
  ```

- Use a model dispatch map (NOT dynamic `prisma[model]` -- see anti-pattern in RESEARCH.md):
  ```typescript
  async function countForModel(model: QuestModel, where: Record<string, any>): Promise<number> {
    switch (model) {
      case "KalenderEintrag": return prisma.kalenderEintrag.count({ where });
      case "Ticket":          return prisma.ticket.count({ where });
      case "Rechnung":        return prisma.rechnung.count({ where });
      case "AktenActivity":   return prisma.aktenActivity.count({ where });
      default: throw new Error(`Unknown quest model: ${model}`);
    }
  }
  ```

- Return `{ current: count, target: condition.count, completed: count >= condition.count }`.

PITFALL AVOIDANCE: For "Die Siegel des Tages" (Fristen), the condition uses `dateField: "erledigtAm"` NOT `datum`. The `datum` field is the deadline date, `erledigtAm` is when it was completed. See Pitfall 1 in RESEARCH.md.

**File 2: `src/lib/gamification/quest-service.ts`**

Create the quest check orchestrator:

1. `checkQuestsForUser(userId: string): Promise<void>` -- Main entry point called by BullMQ or fire-and-forget:
   - Check if user has gamificationOptIn = true. If not, return early.
   - Load user's GameProfile (getOrCreateGameProfile if first time).
   - Load all active DAILY quests: `prisma.quest.findMany({ where: { typ: "DAILY", aktiv: true } })`.
   - For each quest, call `evaluateQuestCondition(quest.bedingung as QuestCondition, userId)`.
   - For completed quests, check for existing QuestCompletion today (dedup!):
     ```typescript
     const today = startOfDay(new Date());
     const existing = await prisma.questCompletion.findFirst({
       where: { userId, questId: quest.id, completedAt: { gte: today } },
     });
     if (existing) continue; // Already awarded today
     ```
   - If not already awarded, calculate streak multiplier and award rewards:
     ```typescript
     const streakMultiplier = getStreakMultiplier(profile.streakTage);
     await awardRewards(userId, quest.xpBelohnung, quest.runenBelohnung, streakMultiplier);
     await prisma.questCompletion.create({
       data: {
         userId, questId: quest.id,
         xpVerdient: quest.xpBelohnung,
         runenVerdient: Math.round(quest.runenBelohnung * streakMultiplier),
       },
     });
     ```
   - After processing all quests, update streak: `await updateStreak(userId)`.

2. `enqueueQuestCheck(userId: string): void` -- Fire-and-forget helper that adds a job to the BullMQ gamification queue. Import the queue from queues.ts (Plan 03 creates it, but the export can be created now with a forward-compatible pattern). For now, implement as a direct call wrapped in `.catch(() => {})` as a fallback until Plan 03 wires BullMQ:
   ```typescript
   export function enqueueQuestCheck(userId: string): void {
     // Will be replaced by BullMQ queue.add() in Plan 03
     checkQuestsForUser(userId).catch(() => {
       // Fire-and-forget: never block business logic
     });
   }
   ```

PITFALL AVOIDANCE: Always check QuestCompletion before awarding to prevent double-counting (Pitfall 5 in RESEARCH.md). The nightly safety net (Plan 03) uses the same checkQuestsForUser which has this dedup built in.
