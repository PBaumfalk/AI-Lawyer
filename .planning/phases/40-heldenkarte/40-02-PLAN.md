**1. Heldenkarte page (src/app/(dashboard)/heldenkarte/page.tsx):**

Create a client component following the exact self-fetching absent-until-loaded pattern from shop/page.tsx.

```typescript
"use client";
```

**State:**
- `data: HeldenkartResponse | null` (null until loaded)
- `loaded: boolean` (false until first fetch completes)
- `page: number` (starts at 1, for quest history pagination)

**Fetch logic:**
- `fetchData` as useCallback that fetches `/api/gamification/heldenkarte?page=${page}`
- On 404: setLoaded(true), return (opted-out user)
- On success: setData(response), setLoaded(true)
- useEffect depends on [fetchData]

**Page change handler:**
- When user clicks prev/next in QuestHistoryTable, update `page` state
- The fetchData callback depends on `page`, so it will re-fetch with new page
- IMPORTANT: To avoid re-running badge evaluation on every page change, consider caching the profile/badge portion of the response. Simple approach: just re-fetch entirely since badge evaluation is fast and caches earned badges in DB.

**Render:**
- Loading: Loader2 spinner centered (same as shop page)
- Not opted in (data is null after load): return null
- Loaded with data:
  - Container: `<div className="p-6 max-w-5xl mx-auto space-y-8">`
  - Page title: `<h1 className="text-2xl font-bold">Heldenkarte</h1>`
  - `<HeroCard {...data.profile} equippedCosmetics={data.equippedCosmetics} />`
  - `<BadgeShowcase badges={data.badges} />`
  - `<QuestHistoryTable items={data.questHistory.items} total={data.questHistory.total} page={data.questHistory.page} pageSize={data.questHistory.pageSize} onPageChange={setPage} />`

Single scroll page layout (no tabs) per user decision in CONTEXT.md.

Define a local TypeScript interface for the API response shape (HeldenkartResponse) at the top of the file, mirroring the API response from Plan 01.
