- Plain text: Same content without HTML formatting, deep-link as full URL
- Footer: "Diese E-Mail wurde automatisch versendet. Bitte antworten Sie nicht auf diese E-Mail."

**Subject lines per type:**
- `neue-nachricht`: "[{kanzleiName}] Neue Nachricht zu Ihrem Verfahren"
- `neues-dokument`: "[{kanzleiName}] Neues Dokument in Ihrem Portal verfuegbar"
- `sachstand-update`: "[{kanzleiName}] Aktualisierung zu Ihrem Verfahren"

Keep HTML simple and inline-styled (no external CSS) for maximum email client compatibility. Use a clean, professional layout with the Kanzlei name as header.

**Step 2: Create src/lib/portal/portal-notification.ts**

This file contains the job data type, the dispatch function, and the processor.

```typescript
export interface PortalNotificationJobData {
  type: "neue-nachricht" | "neues-dokument" | "sachstand-update";
  mandantKontaktId: string;   // Kontakt.id to look up email + einwilligungEmail
  akteId: string;
  deepLinkPath: string;       // e.g. "/akten/{akteId}/nachrichten"
}
```

**enqueuePortalNotification(data: PortalNotificationJobData):**
- Import `portalNotificationQueue` from `@/lib/queue/queues`
- Add job with name `portal-email-{type}` and deduplication:
  - jobId = `portal-${type}-${mandantKontaktId}-${akteId}-${dateKey}` where dateKey = YYYY-MM-DD
  - This prevents duplicate emails for the same event type + mandant + akte within 24h (throttling per user decision)
- Log the enqueue with createLogger("portal-notification")

**processPortalNotification(job data):**
1. Look up Kontakt by mandantKontaktId — get email and einwilligungEmail
2. **DSGVO gate**: If `einwilligungEmail === false`, log and skip (never send)
3. If no email on Kontakt, log and skip
4. Check `isEmailConfigured()` — if false, log and skip
5. Get kanzleiName from SystemSetting (key "kanzlei.name") via getSettingTyped, fallback to process.env.SMTP_FROM or "Kanzlei"
6. Get portalBaseUrl from process.env.NEXTAUTH_URL + "/portal" (or dedicated PORTAL_BASE_URL env)
7. Call `renderPortalEmail()` with the template input
8. Call `sendEmail()` with the rendered output
9. Log success/failure via createLogger
10. Call `logAuditEvent()` to record the email notification (action: "portal:email-sent", resource: mandantKontaktId)

Import dependencies:
- `sendEmail, isEmailConfigured` from `@/lib/email/send`
- `renderPortalEmail` from `@/lib/portal/email-templates`
- `getSettingTyped` from `@/lib/settings/service`
- `logAuditEvent` from `@/lib/audit`
- `prisma` from `@/lib/db`

**Step 3: Register worker in src/worker.ts**

Add a new `portal-notification` worker section following the existing pattern:

```typescript
import { processPortalNotification, type PortalNotificationJobData } from "@/lib/portal/portal-notification";

const portalNotificationWorker = new Worker<PortalNotificationJobData>(
  "portal-notification",
  async (job) => processPortalNotification(job.data),
  {
    connection,
    concurrency: 2,  // Lightweight email dispatch, allow 2 concurrent
    settings: {
      backoffStrategy: (attemptsMade: number) => calculateBackoff(attemptsMade),
    },
  }
);

portalNotificationWorker.on("completed", (job) => {
  if (!job) return;
  log.info({ jobId: job.id, type: job.data.type }, "Portal notification sent");
});

portalNotificationWorker.on("failed", (job, err) => {
  if (!job) return;
  log.error({ jobId: job.id, type: job.data.type, err: err.message }, "Portal notification failed");
});

portalNotificationWorker.on("error", (err) => {
  log.error({ err }, "Portal notification worker error");
});

workers.push(portalNotificationWorker);
log.info("[Worker] portal-notification processor registered");
```
