## Critical Integration Gap

### Root Cause: `(portal)/layout.tsx` Auth Guard Wraps Public Pages

**Severity:** CRITICAL — blocks entire portal for unauthenticated users

**Files involved:**
- `src/app/(portal)/layout.tsx` (lines 17-21) — auth guard
- `src/app/(portal)/login/page.tsx` — login page (BLOCKED)
- `src/app/(portal)/activate/page.tsx` — account activation (BLOCKED)
- `src/app/(portal)/passwort-vergessen/page.tsx` — password reset request (BLOCKED)
- `src/app/(portal)/passwort-reset/page.tsx` — password reset confirm (BLOCKED)

**Mechanism:**

```
User visits /portal/login (unauthenticated)
  → (portal)/layout.tsx runs
    → auth() returns null (no JWT cookie)
    → !session?.user === true
    → redirect("/portal/login")
      → same layout runs again → infinite redirect loop
        → Browser: ERR_TOO_MANY_REDIRECTS
```

The middleware correctly excludes these paths from NextAuth's auth middleware, but the layout's own auth guard is a separate server-side check that runs independently.

**Impact:** The entire Mandantenportal is inaccessible to unauthenticated users. No Mandant can log in, activate their account, or reset their password. This blocks the invite-to-onboarding flow, effectively making all 25 requirements non-functional at runtime.

**Why not caught:** Per-phase static verification confirmed each page's implementation independently. The layout guard was verified as working (Phase 43), and the auth pages were verified as implemented (Phase 44). The cross-phase interaction was flagged as "human verification required" but never tested in a live environment.

**Fix options:**
1. Move `login`, `activate`, `passwort-vergessen`, `passwort-reset` outside the `(portal)` route group (e.g., to `src/app/portal/login/page.tsx` without parentheses)
2. Add path-based guard bypass in `layout.tsx` using `headers()` to read the current URL
